# Principles（按优先级排序）

## P1: 完成 = 主动验证且没有失败项

Agent 不能因为代码写完、看起来合理或没有发现明显问题就声称完成。完成必须满足：测试通过、关键验收标准有 pass/fail 记录，并且至少覆盖主要失败路径。

判定标准：

- 是否运行了与任务相关的测试或验证命令。
- 是否记录了至少 3 类边界或异常场景。
- 是否所有验收标准都有明确 pass/fail。
- 如果有 fail 项，是否没有声称 done。

如果删掉这条 principle，Agent 容易只做 happy path，然后过早宣布完成。

## P2: 宁可进入待确认，也不让脏数据进入正式学习链路

错题本的核心资产是题目、知识点和复习计划。一旦低质量识别结果进入正式链路，会污染后续检索、复习和掌握判断。

判定标准：

- 低置信度识别是否进入 `needs_confirmation`。
- 缺失核心字段时是否阻断成功返回。
- AI 输出是否先经过 schema 校验再进入业务层。
- 用户确认内容是否优先于自动识别结果。

如果删掉这条 principle，Agent 容易为了“流程跑通”而接受不可靠识别结果。

## P3: 不可逆决策必须留下理由和代价

数据模型、AI provider、接口契约、用户数据处理方式等决策会带来迁移成本，必须记录为什么这样选、代价是什么、什么时候需要重新评估。

判定标准：

- 新增不可逆决策时是否写入 ADR。
- ADR 是否包含理由、代价、不可逆原因和判定标准。
- 代码或接口变更是否同步更新相关 Knowledge 文档。

如果删掉这条 principle，Agent 容易一边写代码一边做架构决策，后续无法追溯。

## P4: 契约和命名必须让陌生人秒懂意图

接口字段、函数名、文件名和规则名必须表达业务含义，而不是只表达实现动作。

判定标准：

- 名字是否能说明对象或行为的业务意图。
- 是否避免 `handle_stuff`、`process_data`、`tmp` 这类泛化命名。
- 接口字段是否有明确类型、含义和取值范围。
- 新增命名是否能被不熟悉项目的人理解。

如果删掉这条 principle，Agent 容易写出短期能跑、长期难维护的代码和文档。
