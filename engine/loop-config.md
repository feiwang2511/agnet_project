# Loop Engineering Configuration

本文件定义 AgentOS 在什么条件下可以自动循环执行，什么时候必须停止或升级。

## 停止条件

以下任一条件满足时，任务正常结束：

| 条件 | 说明 |
| --- | --- |
| G4 通过 | 所有 AC pass，主动破坏 >= 5 种，无 fail 项 |
| 超时 | 单任务执行时间超过 60 分钟 |
| 成本上限 | 单任务 token 消耗超过 200k output tokens |
| Eval 阈值 | eval 分数低于 60% 时停止自动循环 |

## 熔断条件

以下任一条件触发时，必须立即中断并通知用户：

| 条件 | 说明 | 恢复方式 |
| --- | --- | --- |
| 重复失败 | 同一 Gate 连续失败 3 次 | 升级为人工 review |
| 成本异常 | 单 session token 消耗超过 500k | 暂停并报告消耗 |
| 破坏性操作 | 检测到删除数据库、删除用户数据意图 | 阻断并等待确认 |
| API 连续异常 | Bedrock 或外部 API 连续 3 次超时/5xx | 暂停并报告 |
| Secrets 泄漏 | Gate 发现代码中包含真实凭证 | 立即停止，不提交 |

## 成本治理

| 预算类型 | 限额 | 超限动作 |
| --- | --- | --- |
| 单任务 output tokens | 200k | 停止当前任务，报告进度 |
| 单 session output tokens | 500k | 熔断，不再启动新任务 |
| 单次 Bedrock 调用 | 30s timeout | 返回 recognition_unavailable |
| DynamoDB 扫描 | 1000 items | 分页，不做全表扫描 |

## 升级策略

| 触发信号 | 升级动作 |
| --- | --- |
| Gate 连续失败 3 次 | 从 L1 升级到 L2（Agent 互查），告知用户 |
| 不可逆架构决策 | 升级到 L3（人类审批），不自动执行 |
| 新错误类型（corrections 中未见过） | 记录到 corrections.log，不自动蒸馏 |
| 数据模型变更 | 强制 PLAN 阶段，升级 G2 到 L3 |
| 涉及用户隐私数据处理 | 升级到 L3，暂停自动执行 |

## 可自动循环的任务

| 任务类型 | 允许自动 loop | 条件 |
| --- | --- | --- |
| bugfix | 是 | 根因明确、影响范围小、有回归测试 |
| hotfix | 是 | 修复点明确、紧急 |
| feature | 否 | 必须人工确认 EVALUATE 和 PLAN |
| refactor | 有条件 | 行为不变边界明确时可自动 |

## Loop 执行流程

```text
开始任务
  → 选择 Profile
  → 执行当前阶段
  → 运行出口 Gate
    → 通过 → 进入下一阶段
    → 失败 → 检查是否可修复
      → 可修复 → 修复后重试（检查熔断）
      → 不可修复 → 升级给人
  → G4 通过 → Done
  → 触发停止/熔断 → 暂停并报告
```

## 渐进放权路径

| 阶段 | 信任度 | 自动范围 |
| --- | --- | --- |
| 初期（当前） | 低 | 只允许 G3 自动检查，其余人工确认 |
| 30 天后 | 中 | bugfix 可全自动，feature 的 BUILD+VERIFY 可自动 |
| 60 天后 | 中高 | bugfix + refactor 全自动，feature PLAN 可 L2 自查 |
| 90 天后 | 高 | 基于 eval 分数动态调整信任边界 |

## 观察指标

定期检查以下指标决定是否调整 Loop 配置：

- Gate 通过率（目标 > 80%）
- 同类 correction 重复率（目标 < 2 次/类）
- Eval 分数趋势（目标持续 >= 80%）
- 人工升级频率（目标递减）
- 任务完成时间趋势（目标稳定或递减）
