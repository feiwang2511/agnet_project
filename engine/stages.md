# Delivery Engine - Stages

## 阶段序列

```text
EVALUATE -> PLAN -> BUILD -> VERIFY
```

这四个阶段的目的不是制造流程感，而是在 Agent 最容易犯错的地方设置停顿：

1. 不理解任务时，不允许直接写方案。
2. 方案不完整时，不允许直接写代码。
3. 实现时，不允许临时做方向性决策。
4. 没有主动验证时，不允许声称完成。

## Stage: EVALUATE

**目的**：确认任务理解正确，避免在错误方向上加速。

**入口条件**：

- 用户给出任务描述。
- Knowledge 已通过 `hooks/on-session-start.sh` 注入。
- Agent 已读取当前 `knowledge/PROJECT.md` 和相关产品/技术约束。

**必须回答的问题**：

- 用户真正要解决的问题是什么？
- 可判定的验收标准（AC）是什么？
- 任务是否涉及不可逆决策？
- 当前 Knowledge 中有哪些规则会影响这个任务？
- 有没有需要先向用户确认的歧义？

**产出物**：

- `spec/<task-id>/evaluate.md`

**产出物最低内容**：

- 任务摘要。
- 3 条以上可判定 AC；bugfix 可降为 1 条以上。
- 相关 Knowledge 引用。
- 风险和不可逆决策清单。
- 是否需要用户确认。

**出口 Gate**：G1: EVALUATE -> PLAN

**预估时间**：5-10 分钟。

**不能做的事**：

- 不写业务代码。
- 不做架构承诺。
- 不跳过 AC 直接进入方案。

## Stage: PLAN

**目的**：在实现前形成可执行方案，尤其是在不可逆决策前停下来。

**入口条件**：

- G1 通过。
- `spec/<task-id>/evaluate.md` 已存在。
- AC 和风险已经明确。

**必须回答的问题**：

- 每条 AC 对应哪些实现点和验证点？
- 是否需要新增或更新 ADR？
- 数据模型、接口契约、错误处理和测试策略是什么？
- 哪些风险需要缓解？
- 哪些事情必须保持不变？

**产出物**：

- `spec/<task-id>/plan.md`

**产出物最低内容**：

- 实现方案。
- AC -> 实现 -> 验证 的映射表。
- 文件或模块边界。
- 错误处理策略。
- 测试策略。
- 新增或更新 ADR 的说明。

**出口 Gate**：G2: PLAN -> BUILD

**预估时间**：10-20 分钟。

**不能做的事**：

- 不写正式业务实现。
- 不把未记录的不可逆决策带入 BUILD。
- 不用“后面再补测试”作为计划。

## Stage: BUILD

**目的**：按已通过的 plan 实现代码和测试。

**入口条件**：

- G2 通过。
- `spec/<task-id>/plan.md` 已存在。
- 方向性决策已经完成。

**必须回答的问题**：

- 当前改动是否严格对应 plan？
- 是否同时更新了测试、文档或契约？
- 是否遵守 `knowledge/TECH.md` 和 active rules？
- 如果发现 plan 有问题，是否应该回退 PLAN？

**产出物**：

- 代码实现。
- 测试。
- 必要的文档或 ADR 更新。

**出口 Gate**：G3: BUILD -> VERIFY

**预估时间**：取决于任务复杂度。

**不能做的事**：

- 不在 BUILD 阶段偷偷改变架构方向。
- 不绕过 schema 或安全规则。
- 不无测试进入 VERIFY。

## Stage: VERIFY

**目的**：证明功能真的满足 AC，并主动尝试破坏它。

**入口条件**：

- G3 通过。
- 代码和测试已经存在。
- 基础质量 Gate 已通过。

**必须回答的问题**：

- 每条 AC 是 pass 还是 fail？
- 哪些失败路径被主动验证过？
- 是否至少覆盖 5 种破坏尝试？
- 是否发现需要回退 BUILD 或 PLAN 的问题？
- 是否可以根据证据声称完成？

**产出物**：

- `spec/<task-id>/verify.md`

**产出物最低内容**：

- AC pass/fail 表。
- 测试命令和结果。
- 至少 5 种主动破坏尝试。
- 未解决风险。
- 是否允许进入 Done。

**出口 Gate**：G4: VERIFY -> Done

**预估时间**：10-15 分钟。

**不能做的事**：

- 不只验证 happy path。
- 不在有 fail 项时声称 done。
- 不用“看起来没问题”替代验证证据。

## 阶段拆分原则

- 两个阶段之间找不到不可逆边界时，可以考虑合并。
- 一个阶段内部出现不可逆决策时，应考虑拆分或回退上一阶段。
- 阶段越多，治理成本越高；阶段越少，Agent 越容易跳步。
- 当前 AgentOS 默认使用四阶段，是为了学习和可控性，而不是为了追求最短路径。
