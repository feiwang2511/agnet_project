# Delivery Engine - Gates

Gate 是阶段之间的出口检查。它不负责完成任务，只负责回答：“现在是否有足够证据进入下一阶段？”

## Gate 级别

| 级别 | 谁判定 | 适用场景 |
| --- | --- | --- |
| L1 | Agent 自查或自动脚本 | 格式、数量、测试、lint、secrets 等可机械判断的内容 |
| L2 | Agent 互查或 devil's advocate | 方案质量、风险完整性、AC 覆盖度 |
| L3 | 人类审批 | 不可逆架构决策、数据迁移、用户数据处理策略 |

## G1: EVALUATE -> PLAN

**当前级别**：L1

**目的**：确认任务已经被正确理解，可以开始设计方案。

**判定条件**：

- [ ] `spec/<task-id>/evaluate.md` 存在。
- [ ] 任务摘要和用户目标一致。
- [ ] feature 任务至少有 3 条可判定 AC。
- [ ] bugfix 任务至少有 1 条可判定 AC。
- [ ] 已列出相关 Knowledge 引用。
- [ ] 已声明是否存在不可逆决策。
- [ ] 有歧义时，不假装清楚，而是记录待确认问题。

**失败处理**：

- AC 不可判定：回到 EVALUATE，重写 AC。
- Knowledge 引用缺失：补读 `knowledge/*` 和 `governance/*`。
- 重大歧义未确认：暂停并询问用户。

**升级条件**：

- 同类 AC 缺失连续 3 次：升级为 L2，由另一个 Agent 或人工 review。
- 涉及用户数据删除、迁移、权限变更：升级为 L3。

**降级条件**：

- 连续 5 次 feature 任务 G1 通过且无后续返工：保持 L1，允许更快通过。

## G2: PLAN -> BUILD

**当前级别**：L2

**目的**：确认方案覆盖 AC 和风险，且不可逆决策已经记录。

**判定条件**：

- [ ] `spec/<task-id>/plan.md` 存在。
- [ ] 每条 AC 都映射到实现点和验证点。
- [ ] 错误处理策略覆盖主要失败路径。
- [ ] 测试策略明确，不能只写“后面补测试”。
- [ ] 新增或改变不可逆决策时，已更新 `knowledge/TECH.md` 的 ADR。
- [ ] 没有违反 active rules。
- [ ] devil's advocate review 没有发现致命缺陷。

**失败处理**：

- AC 覆盖不完整：回到 PLAN 补映射。
- 风险或错误处理缺失：回到 PLAN。
- 不可逆决策未记录：补 ADR 后重新跑 G2。
- 发现方向错误：回退 EVALUATE。

**升级条件**：

- 涉及数据库模型迁移、AI provider 更换、权限模型改变：升级为 L3。
- 同一 plan 连续 3 次被 G2 打回：升级为 L3。

**降级条件**：

- 同一任务类型连续 5 次 G2 通过，且 VERIFY 无重大返工：可降为 L1。

## G3: BUILD -> VERIFY

**当前级别**：L1

**目的**：确认实现和测试达到进入主动验证的最低门槛。

**判定条件**：

- [ ] 实现文件存在，且对应 plan。
- [ ] 测试文件存在。
- [ ] 相关测试通过。
- [ ] `governance/gates/check-basic-quality.sh` 通过。
- [ ] 没有明显 secrets 或隐私数据泄漏。
- [ ] 如果改了接口契约，已同步更新 Knowledge 或 spec。

**失败处理**：

- 测试失败：原地修复。
- 基础质量 Gate 失败：原地修复。
- 实现偏离 plan：回退 PLAN。
- 同一问题 3 次失败：回退 PLAN，重新设计。

**升级条件**：

- 连续 3 次同类测试失败：升级为 L2 review。
- 发现 secrets 或用户隐私泄漏：立即升级 L3 并停止自动流程。

**降级条件**：

- G3 属于自动化检查，默认保持 L1。

## G4: VERIFY -> Done

**当前级别**：L1

**目的**：确认所有 AC 都有证据支持，且主动破坏没有发现未处理失败。

**判定条件**：

- [ ] `spec/<task-id>/verify.md` 存在。
- [ ] 所有 AC 都有 pass/fail 标记。
- [ ] 无 fail 项。
- [ ] 测试命令和结果被记录。
- [ ] 至少 5 种主动破坏尝试被记录。
- [ ] 未解决风险已明确说明，不被包装成完成。

**失败处理**：

- 有 fail 项：回退 BUILD 修复。
- 主动破坏不足 5 种：继续 VERIFY。
- 发现方案层问题：回退 PLAN。
- 无法验证：暂停并说明阻塞。

**升级条件**：

- VERIFY 发现数据污染、隐私泄漏、不可逆变更风险：升级为 L3。
- 同一 AC 连续 3 次 fail：升级为 L2 review。

**降级条件**：

- VERIFY 是完成度底线，不降级为跳过。

## Gate 运行原则

- Gate 失败不是坏事，它是在更便宜的位置发现问题。
- Gate 只能基于证据通过，不能基于“看起来合理”通过。
- Gate 条件应尽量可判定；无法机械判定的条件，应升级到 L2 或 L3。
- 当 Rule 被反复违反时，优先考虑把它下沉为 Gate。
